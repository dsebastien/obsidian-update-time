# 2026-06-24

## Reported problem

User on v1.2.0 (the #7 debounce release) reports the plugin does "many more needless file updates than before." Setting **Save delay** to 120s did not help — it made it worse. User also uses **Syncthing** and suspects external "touches" (files changed on disk without the user typing) trigger spurious `updated` writes that then bounce around between devices.

## Diagnosis

Two compounding problems, one shared root cause: the plugin reacts to *"the file was modified"* (an `mtime`/event) instead of *"the meaningful content changed."*

### Problem 1 — self-feeding write loop (regression from #7)

`processFile` writes `updated = mtime` read when the debounce timer fires, but the write itself bumps the real `mtime` forward by `saveDelayInSeconds`, then fires another `modify`:

1. User stops at `t₀`; `mtime = t₀`.
2. After the delay, plugin writes `updated = t₀`; the write bumps `mtime → t₀ + delay` and fires `modify`.
3. Next cycle reads `mtime = t₀ + delay` and compares to the just-written `updated` (`t₀`) via `isAfter(mtime, updated + MINUTES_BETWEEN_SAVES)`. `MINUTES_BETWEEN_SAVES` is hardcoded to **1 min** (`constants.ts:10`).
4. Because the delay (120s) **exceeds** that 1-min window, the condition is always true → writes again → bumps `mtime` again → repeats forever, ~every `saveDelayInSeconds`, on every file ever edited.

The loop only sustains when `saveDelayInSeconds > 60s`; the default 2s self-terminates. Raising the delay to reduce writes is exactly what triggers the runaway. The old version stayed stable because it wrote within the same minute as the edit (identical minute-truncated bytes → no further change).

### Problem 2 — touches indistinguishable from edits (Syncthing)

`vault.on('modify')` fires for any byte/`mtime` change: typing, AI tools, the plugin's own writes, **and Syncthing rewriting files on disk**. The plugin treats all as edits, so a Syncthing touch → `updated` bump → genuine content change → Syncthing propagates → `modify` on the other device → bump → cross-device ping-pong.

## Shipped now — base-logic stabilization (Problem 1)

Self-induced-write detection. Strictly prevents the plugin from reacting to its own write, regardless of save delay. Chosen over the user's "derive the value window from the delay" idea because it fixes the cause directly instead of masking it with a larger churn window, and keeps `updated` churn semantics intact.

- `src/app/utils/is-self-induced-modify.fn.ts` — pure predicate: `true` when a recorded post-write `mtime` equals the current `mtime` (the event is our own echo).
- `src/app/utils/is-self-induced-modify.fn.spec.ts` — 4 specs.
- `src/app/plugin.ts`:
    - New `lastWriteMtimes: Map<path, number>`.
    - After every successful `processFrontMatter` write, record `file.stat.mtime`.
    - At the start of `processFile`, consume the recorded entry once; if it matches the current `mtime`, return without processing. Consuming once means the next genuine change is still handled. Timing is safe: the post-write `.set` runs on the await-continuation microtask, before the debounced `processFile` macrotask of the echo event.
    - `onunload` / `saveSettings` clear the map alongside the debouncers.
- `documentation/Business Rules.md` — new invariant #14.

## Deferred — Problem 2 (touches vs. real edits)

Tracked as [issue #8](https://github.com/dsebastien/obsidian-update-time/issues/8) + plan (`documentation/plans/distinguish-real-edits-from-touches.md`). User picked the **hybrid** approach (`editor-change` positive signal + content-signature for external/AI writes) but flagged real constraints: content hashing is costly if done broadly, in-memory baselines go stale, and bulk external changes (close Obsidian → modify 5K files → reopen) must not trigger a hash storm. The plan addresses these: never precompute hashes across the vault, only disambiguate files that actively fire `modify`, prefer the cheap `editor-change` signal, and fall back to the existing `mtime`-vs-`updated` heuristic when no baseline exists (the #14 self-write guard keeps that fallback loop-free).
