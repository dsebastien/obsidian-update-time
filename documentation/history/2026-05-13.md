## Decisions

- **Trim + fallback to defaults** for invalid input. Empty / whitespace-only / missing values resolve to `PROPERTY_CREATED` / `PROPERTY_UPDATED` at write time. No error UI ‚Äî keeps the settings UX simple and matches how the existing `ignoredFolders` field tolerates missing/invalid data on load.
- **No migration of existing notes** when a property name changes. The plugin only affects future writes. Documented in both `docs/configuration.md` and the per-field description in the settings tab.
- **No business-rule change required.** Invariants #2 and #3 (preserve user `created`; debounce `updated`) are about property values, not keys; they still hold when the key is renamed.

## Issue #5 ‚Äî batch backfill command

### Changes

- `src/app/utils/apply-backfill-to-front-matter.fn.ts` ‚Äî new pure helper. "Fill missing only" mutator:
    - Writes `createdKey` only when missing (business rule #2 ‚Äî never overwrite a user-provided created value, even if invalid).
    - Writes `updatedKey` when missing OR unparsable (matches the live handler's first-write branch).
    - Returns whether the front-matter object was mutated.
- `src/app/utils/apply-backfill-to-front-matter.fn.spec.ts` ‚Äî unit tests covering both keys missing, valid existing values, invalid existing values, and custom property names.
- `src/app/ui/backfill-confirm-modal.ts` ‚Äî new `Modal` shown before the command runs. Lists what will happen (file count, what is skipped, that existing values won't be touched) and offers Cancel / Run buttons.
- `src/app/commands/backfill-properties.ts` ‚Äî new module:
    - `registerBackfillPropertiesCommand(plugin)` registers the command with `plugin.addCommand`.
    - `runBackfillProperties(plugin)` iterates `app.vault.getMarkdownFiles()`, filters via `plugin.shouldFileBeIgnored`, parses `ctime` / `mtime`, calls `applyBackfillToFrontMatter` inside `processFrontMatter`, and tracks `BackfillResult` counters (`total`, `processed`, `updated`, `skipped`, `errors`).
    - Brackets the run with two `Notice` toasts.
    - `YAMLParseError` and any other write failure increment `errors` and the loop continues with the next file.
- `src/app/commands/index.ts` ‚Äî new `registerCommands(plugin)` aggregator.
- `src/app/plugin.ts` ‚Äî `onload` now calls `registerCommands(this)`.

### Documentation

- `README.md` ‚Äî added "One-shot backfill" feature bullet.
- `docs/README.md` ‚Äî added the same bullet; updated the "What the plugin accesses" section to mention the on-demand vault iteration.
- `docs/usage.md` ‚Äî new "Backfilling existing notes" section explaining how to run the command, what it does, and the backup recommendation.
- `documentation/Architecture.md` ‚Äî added the new files to "Entry point & plugin class"; new "Commands" section; added the new helper to "Helpers".
- `documentation/Configuration.md` ‚Äî new "Commands" section.
- `documentation/Domain Model.md` ‚Äî added `BackfillResult` and `ApplyBackfillArgs` types.
- `documentation/Business Rules.md` ‚Äî new invariant #11 covering the batch path (confirmation modal + same filters + same overwrite rules as the live handler; debounce is the only thing the batch is allowed to bypass).

### Decisions

- **Fill missing only.** Matches the issue's exact request; safest semantics; reuses business rule #2 as-is.
- **Reuse `shouldFileBeIgnored` from the plugin instance.** Avoids divergence between live and batch paths.
- **Confirmation modal** before running. Vault-wide writes are a big enough blast radius to warrant an explicit confirm; matches the executing-actions-with-care guidance.
- **Start + end `Notice`** for progress. Avoids the complexity of a live progress modal; matches the existing plugin's UI style (no other progress UIs today).
- **`apply-backfill-to-front-matter.fn.ts` as a pure helper** so the core logic is unit-testable without mocking Obsidian's `processFrontMatter`.
- **Per-file `try`/`catch`** so a single bad file doesn't abort the whole run.
